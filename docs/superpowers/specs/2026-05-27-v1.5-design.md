# HydrogenEyeCare 1.5 设计说明

## 目标

HydrogenEyeCare 1.5 在现有 1.0 托盘应用基础上做体验增强，但不改变核心的 20 分钟工作、20 秒休息节奏。此次更新增加提醒窗口主题、今日远眺统计、增强版关于窗口、日志轮转，以及谨慎的发布体积优化路径。

稳定性优先级高于体积和新功能。所有改动继续保持在现有 .NET 10 + WinForms 技术栈内，并保留“启动后直接驻留托盘”的产品形态。

## 当前基线

当前应用已经具备：

- 启动后直接驻留 Windows 系统托盘。
- 固定 20 分钟工作计时和 20 秒休息提醒。
- 支持延迟 5 分钟，最多连续延迟 2 次。
- 托盘菜单包含开机自启、静音、暂停/恢复、立即休息、关于、退出。
- 配置保存到 `%LOCALAPPDATA%\HydrogenEyeCare\config.json`。
- 错误日志写入 `%LOCALAPPDATA%\HydrogenEyeCare\error.log`。
- 使用 WinForms 提醒窗口和 Windows 原生托盘菜单。

## 范围

1.5 包含：

- 托盘菜单增加界面主题选择。
- 三套主题：森林绿、迷雾蓝、暗岩灰。
- 只对提醒窗口做颜色主题化，不改变提醒窗口尺寸、布局和交互。
- 关于窗口里的统计区域跟随当前主题做轻量配色。
- 增加内存级今日成功远眺计数。
- 增强关于窗口内容。
- 按大小轮转 `error.log`。
- 尝试安全的发布体积优化，并保留回退路径。

1.5 不包含：

- 全局主题化。
- 自绘 Windows 原生托盘菜单。
- 修改固定的 20 分钟工作、20 秒休息、5 分钟延迟、最多 2 次延迟规则。
- 持久化历史统计。
- NativeAOT。

轻量挂机检测作为 1.5 的后置可选项。只有在主要 1.5 功能完成并验证通过后，才尝试实现。如果实现或测试中发现不稳定、边界复杂或时间成本过高，就延期到 1.6。

## 主题设计

新增应用主题模型，包含三个值：

- `ForestGreen`：森林绿
- `MistBlue`：迷雾蓝
- `RockGray`：暗岩灰

默认主题为森林绿，因为它最符合护眼工具的产品语义。

当前主题保存到 `AppConfig`，程序重启后保留用户上次选择。已有配置文件没有主题字段时，应安全加载并回退到森林绿。

托盘菜单新增原生子菜单：

- `界面主题`
  - `森林绿`
  - `迷雾蓝`
  - `暗岩灰`

同一时间只有一个主题菜单项处于勾选状态。用户选择主题后立即保存配置，并影响之后打开的提醒窗口。如果提醒窗口已经打开，可以保持当前颜色直到关闭；只有在实现简单且风险低时，才考虑即时刷新当前窗口。低风险行为优先。

## 提醒窗口主题化

提醒窗口保留现有尺寸、布局、不抢焦点行为和交互模型。

主题只影响：

- 窗口背景色。
- 边框颜色。
- 标题文字颜色。
- 倒计时数字颜色。
- 提示语文字颜色。
- 延迟按钮的背景、边框和文字颜色。

主题调色板用一个小型本地类型表达，例如 `ThemePalette`。创建 `RestReminderForm` 时把当前主题调色板传进去，避免把 UI 配色逻辑混进计时器逻辑。

## 今日远眺统计

新增一个内存级今日统计组件，用来记录当天成功完成休息的次数。

计数规则：

- 完整度过一次 20 秒休息后，今日成功远眺次数 +1。
- 点击延迟不计数。
- 通过 `立即休息` 手动触发休息后，如果完整度过 20 秒，也计数。
- 统计只保存在内存中。
- 程序退出后清零。
- 跨零点后清零。

统计组件自己维护当前日期和计数。每次读取或增加计数前，先检查系统日期是否变化；如果已经跨天，则把计数重置为 0。

## 关于窗口

增强关于窗口，显示：

- 产品名。
- 从程序集元数据读取的当前版本。
- GitHub 链接：`https://github.com/zhirenc13-lab/HydrogenEyeCare`
- 今日成功远眺次数。
- 根据今日次数生成的一句动态评语。

建议评语分段：

- 0 次：`今天还没有完成远眺，先从下一次提醒开始。`
- 1-3 次：`今天已经开始照顾眼睛了。`
- 4-7 次：`今天节奏不错，继续保持。`
- 8 次及以上：`今天护眼执行得很稳定，注意也要离开座位活动一下。`

关于窗口只让统计和评语区域使用当前主题的轻量配色。其余区域保持简洁、系统化风格。

## 日志轮转

每次写入异常到 `error.log` 前，先检查当前日志文件大小。

如果 `error.log` 大于 5 MB：

- 写入新日志前先清空该文件。
- 继续使用同一个 `error.log` 路径。
- 不创建历史归档文件。

如果日志轮转失败，日志组件不应继续向外抛异常。日志失败不能导致应用崩溃或影响正常使用。

## 轻量挂机检测

轻量挂机检测是 1.5 的第二优先级。

目标行为：

- 使用 `GetLastInputInfo` 检测键盘和鼠标无输入是否达到 3 分钟。
- 在工作计时期间，如果用户已经 3 分钟无输入，则挂起工作计时。
- 用户回来后，重置为一个新的 20 分钟工作周期。

这部分逻辑会触碰计时状态机，必须谨慎测试。如果主要 1.5 功能还没有全部验证完成，或者挂机检测出现不清晰的边界情况，就延期到 1.6。

## 发布体积优化

不使用 NativeAOT。

发布流程可以尝试 trimming 作为安全优化，但只有在手动验证通过后，trimming 产物才可以作为发布候选。必须保留稳定的自包含单文件发布路径作为回退方案。

trimming 构建的手动验证至少包括：

- 应用能启动并进入托盘。
- 托盘菜单能打开。
- 主题可以切换。
- 立即休息能打开提醒窗口。
- 延迟功能正常。
- 完成休息后统计会增加。
- 关于窗口能打开并显示正确内容。
- 应用能正常退出并重新启动。

如果 trimming 构建破坏任何核心行为，则使用 1.0 风格的稳定自包含单文件配置发布。

## 测试策略

实现时尽量加入聚焦的测试：

- 主题配置的默认值和持久化。
- 今日统计的递增和跨零点清零。
- 日志轮转阈值行为。

WinForms UI 行为必须做手动运行验证：

- 托盘启动。
- 主题子菜单选择。
- 三套主题下的提醒窗口颜色。
- 关于窗口内容和主题化统计区域。
- 休息完成后的计数增加。
- 日志轮转不影响应用行为。
- 发布产物冒烟测试，尤其是启用 trimming 时。

## 成功标准

1. 用户可以从托盘菜单选择森林绿、迷雾蓝或暗岩灰。
2. 提醒窗口使用当前主题颜色，但形态和行为不变。
3. 关于窗口显示版本、GitHub 链接、今日成功远眺次数和动态评语。
4. 完整休息会增加今日次数，延迟不会增加。
5. 今日次数只保存在内存中，程序重启或跨零点后清零。
6. `error.log` 超过 5 MB 时，在写入新日志前被清空。
7. 发布体积优化只以可回退方式尝试，稳定发布配置保留。
8. 轻量挂机检测要么验证稳定后进入 1.5，要么延期到 1.6。
